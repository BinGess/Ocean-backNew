# Ocean Backend

NestJS + Prisma + PostgreSQL backend for the Ocean Flutter app.

## Scope

- Email/password auth with Argon2id password hashes.
- JWT access tokens and rotating refresh sessions.
- Profile APIs.
- Text and structured-data sync for records, daily summaries, daily moods, weekly/monthly report cache, and legacy weekly insights.
- No audio upload or local audio path persistence.

## Quick Start

```bash
npm install
cp .env.example .env
npm run prisma:generate
npm run prisma:migrate
npm run start:dev
```

Swagger is available at `/docs`.

## Sealos Deployment

For China mainland production, prefer Sealos over Render so the API can run in a domestic availability zone and use a matching ICP filing path.

Prepared deployment files:

- `Dockerfile`: production image for Sealos App Deploy.
- `.dockerignore`: keeps local build output, dependencies, and secrets out of the image context.
- `.github/workflows/publish-image.yml`: publishes `ghcr.io/<OWNER>/<REPO>:latest` and immutable commit tags.
- `sealos.env.example`: runtime environment variable template.
- `docs/SEALOS_DEPLOY.md`: step-by-step Sealos deployment guide.

The Sealos app should expose port `3000`, use `/health` as the health check path, and start with:

```bash
npm run start:sealos
```

In Sealos, leave the startup command empty so the image default command runs this script.

## Render Deployment

This repository includes a Render Blueprint in `render.yaml`. It provisions:

- `ocean-back-new`: Node.js web service in the Singapore region.
- `ocean-postgres`: Render PostgreSQL database.
- `DATABASE_URL`: injected from the Render PostgreSQL internal connection string.
- JWT secrets: generated by Render during Blueprint creation.

The Blueprint uses paid baseline plans (`starter` web service and `basic-256mb` Postgres) because Ocean stores personal data and Render's free Postgres expires after 30 days. For a short-lived demo you can change both plans to `free`, but do not use that for real user data.

### First Deploy

1. Push this project to GitHub, GitLab, or Bitbucket.
2. Open the Render Blueprint page:

   ```text
   https://dashboard.render.com/blueprint/new?repo=<YOUR_HTTPS_GIT_REPO_URL>
   ```

3. Review the resources from `render.yaml` and apply the Blueprint.
4. Wait for the database and web service deploy to finish.
5. Verify the API:

   ```text
   https://<YOUR_RENDER_SERVICE>.onrender.com/health
   https://<YOUR_RENDER_SERVICE>.onrender.com/docs
   ```

### Deploy Commands

Render runs these commands from `render.yaml`:

```bash
npm ci && npm run prisma:generate && npm run build
npm run prisma:deploy
npm run start:prod
```

The pre-deploy command applies Prisma migrations before the new API instance starts. For future schema changes, create a new migration locally and commit it:

```bash
npm run prisma:migrate -- --name describe_change
```

## Sync APIs

- `GET /sync/snapshot` returns full server state and the latest cursor.
- `POST /sync/push` accepts batched local-first changes.
- `GET /sync/pull?cursor=<revision>` returns changes after the cursor.

Conflict behavior is last-write-wins using the client's update timestamp. Deletes are kept as tombstones in incremental pull responses but omitted from snapshots.
